# ADR-004: Preference-pair synthesis — rule-based adversarial perturbation on xLAM subset

**Status:** Accepted
**Date:** 2026-07-18
**Decision:** For v1, generate preference pairs using rule-based adversarial perturbations of the xLAM-source SFT examples only. Skip LLM-generated rejected samples and skip Hermes for the preference stage.

## Context

DPO (Week 6, per ADR-002) requires preference pairs: for each *chosen* correct tool call, a *rejected* plausible-but-wrong alternative that the model should learn to disprefer. We need to build ~10-15K such pairs from our SFT data pool.

Three axes to decide:
1. Which subset of SFT examples get preference pairs?
2. How are the "rejected" examples generated?
3. What ratio (if any) of rejected variants per chosen example?

## Options considered

### Source subset

- **Option A: All 12K post-dedup examples.** Preserves source diversity.
- **Option B: xLAM subset only (~11K).** Skips Hermes.
- **Option C: New sampling.** Curate a fresh subset optimized for adversarial variety.

### Rejected generation method

- **Rule-based:** deterministic perturbations (swap tool name, drop arg, corrupt JSON, hallucinate tool).
- **LLM-generated:** ask Claude/GPT to produce a plausible-but-wrong tool call.
- **Hybrid:** some fraction from each.

### Ratio

- **1:1** — one rejected per chosen (simplest, cheapest, standard).
- **1:N** — multiple rejected variants per chosen (more DPO signal but N× cost).

## Decision

- **Source subset:** Option B — xLAM subset only (~11K).
- **Rejected generation:** rule-based only.
- **Ratio:** 1:1 — one rejected per chosen.

Total preference pairs after this stage: ~11K.

## Rationale

### Why xLAM-only

xLAM examples have a clean, structured 3-turn shape: system prompt with a JSON tools list, a user query, and an assistant response that is a JSON tool call. This structure is *directly perturbable* by rule-based transformations — swap the tool name to another valid tool from the same list, drop a required argument, corrupt JSON syntax.

Hermes examples are multi-turn conversations with tool calls embedded in prose. Reliably parsing "where is the tool call in this assistant message" is brittle. Attempting rule-based perturbation on Hermes would either fail silently or produce low-quality rejected samples. Skipping Hermes for the preference stage is the honest call.

Trade-off: we lose Hermes-style diversity in the DPO training signal. That's acceptable because:
- SFT (Week 4) already exposed the model to Hermes conversational style
- DPO is teaching the model to *avoid* specific failure modes; xLAM's cleaner structure lets us construct sharper "rejected" examples the model can actually learn from
- If DPO quality is weak in Week 6, we can synthesize additional Hermes-style rejected samples via LLM as an upgrade path

### Why rule-based only

Cost economics: LLM-generated rejected samples for 11K pairs would run ~$30-50 in Claude API calls. Rule-based is free.

More importantly, rule-based perturbations are *deterministic* and *documentable*. When an interviewer asks "how did you generate the rejected examples in your preference data?", we can produce a taxonomy of the exact perturbation types and cite counts per type. LLM-generated failures are harder to categorize — the LLM's mistakes vary in ways that are hard to characterize post-hoc.

The known limitation of rule-based: any attacker who reads the perturbation rules can construct adversarial inputs that evade the model's trained-to-reject patterns. This is a real limitation, but the risk model doesn't apply to us — this is a resume-credential fine-tune, not an adversarial-robustness product.

If Week 6 DPO metrics are weak, extending with LLM-generated rejected samples is a well-defined upgrade path (ADR-004a to be filed if we take it).

### Why 1:1 ratio

- Matches TRL's `DPOTrainer` default expectation
- Fits the $1000 budget cleanly (no multiplied cost)
- Multiple rejected variants per chosen is documented in the DPO literature as *sometimes* helpful but the effect size is small compared to base data quality — better spent on data quality than variant count for v1

## Perturbation taxonomy

For each xLAM example (with parsed tool call `{"name": "X", "arguments": {...}}`), the rejected variant is generated by uniformly sampling one of five perturbation types:

1. **wrong_tool_from_list** — replace `name` with a different tool from the same `tools` list in the system prompt. (~20% target rate)
2. **hallucinated_tool** — replace `name` with a plausible-but-nonexistent tool name (e.g., append "_v2" or use a common but not-in-list name). (~20%)
3. **missing_required_arg** — remove one required argument from `arguments`. (~20%)
4. **wrong_arg_value** — replace a string argument's value with a random other string from another xLAM example's arguments, or replace a number with a plausibly-wrong number. (~20%)
5. **malformed_json** — corrupt the JSON with a specific error (unterminated string, trailing comma, missing brace). (~20%)

Even distribution is not strictly enforced — if a specific perturbation can't apply to an example (e.g., no other tools in list), the pipeline falls back to the next perturbation type in a fixed order and records the actual type applied.

Provenance: each preference pair records `perturbation_type` and `perturbation_seed` for reproducibility and Week 6 analysis (which perturbation types the model learned to reject best).

## Consequences

- `data/assemble_preferences.py` implements the 5 perturbation functions plus a dispatcher.
- Preference pairs preserve the same `messages` structure but with two variants: `chosen` (the original assistant response) and `rejected` (the perturbed one).
- Perturbation-type distribution is reported in the script's summary output.
- Week 6 DPO analysis includes per-perturbation-type "reward accuracy" so we can see which failure modes the model learns to reject fastest.
- Model card must document the perturbation taxonomy for reproducibility and honesty about the DPO training signal.

## Alternatives rejected specifically

- **All-source coverage (Option A) with rule-based:** rejected because reliably perturbing Hermes prose examples adds significant parsing complexity for little quality gain.
- **LLM-generated only:** rejected due to cost + reduced documentability of the failure taxonomy.
- **1:N ratio:** deferred — small documented effect size vs. cost multiplier.

## Revisit trigger

Revisit if:
- Week 6 DPO training shows poor "reward accuracy" plateauing well below 90% — indicates rejected samples aren't distinguishable enough from chosen
- Week 7 BFCL eval shows the model still fails on failure modes not covered by our 5 perturbation types
- We decide to publish a research writeup where LLM-generated adversarial rejected samples would strengthen the methodology story
